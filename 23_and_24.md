### §1c BT-Q succession seal vs vanilla recalc — REFRAMED + safe fix
Investigated the seal (QING_princes_seal_succession, se_QING_PRINCES.txt:285). Findings that
narrow the doc's concern:
- The seal grants the vanilla `nominated_heir_modifier` (`support_for_character_as_heir = 50`,
  `duration = -1` = PERMANENT). Vanilla `recalc_succession` RESPECTS support scores, so it re-picks
  the sealed heir automatically as long as the modifier lives. The feared "recalc silently
  overwrites the sealed heir" only happens if another character legitimately exceeds +50 support —
  rare, and arguably correct if it does.
- Oracle answers (from the on_action tree): there is NO `on_recalc_succession` hook, and NO
  registered `on_birth` on_action. So "suppress on the recalc hook" is not available.
- The seal stores NO persistent heir reference (only `save_scope_as = qing_sealed_heir`, which does
  not survive the tick) and the accession (qing_succession.2) reads only the
  `qing_secret_succession_sealed` FLAG.
**The real, boot-independent correctness bug this exposes:** if the sealed heir DIES, the permanent
modifier dies with him but `qing_secret_succession_sealed` stays set (stale). The accession event
then treats the succession as "sealed/smooth" when there is no longer any sealed heir — a genuine
inconsistency. **Fix (safe, additive, idempotent):** new `QING_princes_reassert_seal` (country
scope) — if the seal flag is set but NO living character holds `nominated_heir_modifier`, the seal
is broken: clear the stale flag and reopen the jockeying (set qing_succession_jockeying) so the
accession resolves as disputed (correct) and the player's seal lever works again. Wired into the
EXISTING CHI `on_character_death` hook (qing_mechanics_on_actions.txt) — the moment a ruling-family
death can break the seal. Does NOT touch the recalc path (vanilla's +50 handling is left to do its
job). Static-verified only (no boot).

## Design (chosen)
NEW per-character var `qing_favored_heir` = a REF to the candidate prince this official backs (or unset =
neutral / no contest). Because vanilla's native heir-attraction can't be pointed at the scripted princes,
this is a SCRIPTED attitude surfaced via a scripted/custom-loc card row (not the native glue) — the same
choice the mod already made for qing_char_affinity/qing_char_stance.

- COMPUTE (QING_heirfavor_pulse, new se_QING_HEIRFAVOR.txt, called from QING_princes_pulse AFTER
  recompute_roster): only runs with >=2 contending princes and an UNSEALED succession (a sealed 立儲 ends
  the jockeying, so officials fall in line — clear the var). Rank princes: frontrunner (max backing) vs
  dark-horse (2nd). Each GC office/sub-position holder picks: a LOYAL official (qing_char_affinity high)
  backs the FRONTRUNNER (the throne's man); a DISAFFECTED one (affinity low) backs the DARK-HORSE (the
  rival clique); mid = neutral (unset). Iterate qing_council_members + the sub-position roster lists +
  figurehead seats, all via the proven every_in_list idiom, scoring in each holder's char scope.
- WIRE INTO SUCCESSION: tally council favor per prince; add a modest per-pulse nudge to each prince's
  qing_prince_backing = (#officials backing him) so a council-backed dark-horse can climb and overtake the
  frontrunner -> changes who QING_princes_seal_succession names. A SPLIT council (both princes have backers)
  docks qing_dynastic_harmony -1 (the court divided against itself). Band-gated, clamped, no runaway.
- DISPLAY: clone qing_favor_square -> qing_heir_favor display on each office card name-row, reading
  qing_favored_heir; shows the backed prince's name + a tooltip. Sub-position panels get it on their rows.
- Excludes the princes themselves + the emperor (can't favor an heir over himself), mirroring the favor
  square's emperor exclusion.
