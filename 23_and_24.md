### §1c BT-Q succession seal vs vanilla recalc — REFRAMED + safe fix
Investigated the seal (QING_princes_seal_succession, se_QING_PRINCES.txt:285). Findings that
narrow the doc's concern:
- The seal grants the vanilla `nominated_heir_modifier` (`support_for_character_as_heir = 50`,

  overwrites the sealed heir" only happens if another character legitimately exceeds +50 support —
  rare, and arguably correct if it does.

  registered `on_birth` on_action. So "suppress on the recalc hook" is not available.

  not survive the tick) and the accession (qing_succession.2) reads only the
  `qing_secret_succession_sealed` FLAG.
**The real, boot-independent correctness bug this exposes:** if the sealed heir DIES, the permanent
modifier dies with him but `qing_secret_succession_sealed` stays set (stale). The accession event
then treats the succession as "sealed/smooth" when there is no longer any sealed heir — a genuine
inconsistency. **Fix (safe, additive, idempotent):** new `QING_princes_reassert_seal` (country
scope) — if the seal flag is set but NO living character holds `nominated_heir_modifier`, the seal
is broken: clear the stale flag and reopen the jockeying (set qing_succession_jockeying) so the
accession resolves as disputed (correct) and the player's seal lever works again. Wired into the
EXISTING CHI `on_character_death` hook (qing_mechanics_on_actions.txt) — the moment a ruling-family
death can break the seal. Does NOT touch the recalc path (vanilla's +50 handling is left to do its
job). Static-verified only (no boot).
