### [#163 SPHERE PROBE] Phase-0 capability probe for the four-power dynamic
### sphere-of-influence build. Purpose: settle the ONE unproven engine capability
### the design rides on — can a STATE read a NUMERIC variable off a NEIGHBOURING
### state during iteration (the PULL model), or must we fall back to PUSH?
###
### WHY THE PREVIOUS PROBE WAS INVALID (the reason error.log said
### "sphere_probe_score is set but never used"):
###   The old probe only ever "read" the var inside debug_log strings, e.g.
###       msg = "... ="
###   A [ ... .GetValue ] inside a log string is a GUI DATA-FUNCTION read, evaluated
###   by the interface layer — NOT a script read. So (a) no script trigger or effect
###   ever consumed the variable, which is exactly why the script analyzer flagged it
###   "set but never used"; and (b) the data-function path resolves the same way
###   whether or not the *script* engine can do a cross-state numeric read, so it
###   could never have produced a real PULL-vs-PUSH verdict. It always "passed".
###
### THE FIX: this rewrite CONSUMES the neighbour's variable in genuine SCRIPT
### context — a trigger comparison (var:... >= ...) and an arithmetic effect
### (change_variable add = scope:....var:...). If those work, the accumulated sum is
### non-zero and PULL is supported; the verdict itself is decided by a script trigger
### (if = { limit = { var:... > 0 } }), not by reading a log line.
###
### This file has NO BOM to match its sibling scripted_effects (se_*.txt all start
### with ASCII). Its on_action hook (common/on_action/SPHERE_probe_debug.txt) DOES
### carry a UTF-8 BOM, which every on_action file in this build requires.
###
### Runs once at game start (guarded by a global flag) only when CHI is in the game.
### Extract with:   grep "IMP19C SPHERE" logs/debug.log
### Requires -debug_mode. This whole file + its on_action are throwaway; delete both
### once the PULL-vs-PUSH verdict is recorded. See memory imp19c-sphere-idioms-oracle.

SPHERE_probe_run = {
	# Scope: CHI (country).
	LOG_line = { sys = SPHERE  msg = "PROBE START (CHI) ============================" }

	# --- 1. Source = CHI's capital state. Stamp a sentinel and a zeroed accumulator. ---
	capital_scope = {
		state = {
			save_scope_as = sphere_src_state
			set_variable = { name = sphere_probe_score  value = 42 }
			set_variable = { name = sphere_pull_sum     value = 0 }
			set_variable = { name = sphere_pull_count   value = 0 }
			LOG_line = { sys = SPHERE  msg = "source state = (owner); sentinel stamped" }
		}
	}

	# --- 2. First pass: stamp a KNOWN numeric value (10) on every land neighbour,
	#        reached via the proven walk state -> province -> neighbor_province -> state.
	#        This is a plain write; the capability under test is the READ in step 3. ---
	scope:sphere_src_state = {
		every_state_province = {
			every_neighbor_province = {
				limit = {
					is_sea = no
					total_population > 0
				}
				state = {
					# `limit` filters only inside list iterators / if / while — NOT as a
					# bare guard in a plain scope switch, so wrap it in `if`.
					if = {
						limit = { NOT = { this = scope:sphere_src_state } }
						set_variable = { name = sphere_nbr_val  value = 10 }
					}
				}
			}
		}
	}

	# --- 3. THE ACTUAL SCRIPT-LEVEL TEST. From the source, walk the neighbours again
	#        and CONSUME each neighbour's own numeric var in script context:
	#          (a) TRIGGER read  — `var:sphere_nbr_val >= 10` inside a limit; a
	#              neighbour only survives the filter if the engine can evaluate that
	#              comparison against the neighbour's OWN scope mid-iteration.
	#          (b) EFFECT read   — pull that neighbour's value into the SOURCE's
	#              accumulator via `change_variable add = scope:sphere_cur_nbr.var:...`.
	#        This is the exact operation the contest step needs (read an adjacent
	#        state's influence number and combine it), done the way the real system
	#        would do it — not via a debug_log data-function. ---
	scope:sphere_src_state = {
		every_state_province = {
			every_neighbor_province = {
				limit = {
					is_sea = no
					total_population > 0
				}
				state = {
					# `limit` must live inside `if` here (plain scope switch, not an iterator).
					if = {
						limit = {
							NOT = { this = scope:sphere_src_state }
							has_variable = sphere_nbr_val
							var:sphere_nbr_val >= 10          # (a) script TRIGGER read of the neighbour's own var
						}
						save_scope_as = sphere_cur_nbr
						scope:sphere_src_state = {
							# (b) script EFFECT read: pull the neighbour's numeric var across scopes.
							change_variable = { name = sphere_pull_sum   add = scope:sphere_cur_nbr.var:sphere_nbr_val }
							change_variable = { name = sphere_pull_count  add = 1 }
						}
					}
				}
			}
		}
	}

	# --- 4. VERDICT decided by a SCRIPT trigger on the accumulated sum, not by a log
	#        read. If the cross-state numeric read worked, sum = 10 * neighbours found. ---
	scope:sphere_src_state = {
		if = {
			limit = { var:sphere_pull_sum > 0 }
			LOG_line = { sys = SPHERE  msg = "PULL VERDICT = SUPPORTED. neighbours read = , accumulated sum = (expect 10 x count). Script-level cross-state numeric var read WORKS — build the PULL model." }
		}
		else = {
			LOG_line = { sys = SPHERE  msg = "PULL VERDICT = FAILED. sum still 0 despite neighbours being stamped — script cannot read an adjacent state's numeric var mid-iteration. Fall back to the PUSH model (each source writes into its neighbours)." }
		}

		# --- 5. Cleanup: remove every sentinel so nothing is left "set but never used". ---
		remove_variable = sphere_probe_score
		remove_variable = sphere_pull_sum
		remove_variable = sphere_pull_count
		every_state_province = {
			every_neighbor_province = {
				limit = { is_sea = no  total_population > 0 }
				state = {
					if = {
						limit = { has_variable = sphere_nbr_val }
						remove_variable = sphere_nbr_val
					}
				}
			}
		}
	}

	LOG_line = { sys = SPHERE  msg = "PROBE END ====================================" }
}
